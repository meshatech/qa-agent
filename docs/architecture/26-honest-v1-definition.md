# Honest v1 Definition

## Purpose

Define what a credible, sellable, and technically honest Agent QA v1 means.

The goal of v1 is not to promise that the agent can test any frontend perfectly. The goal is to prove a safe production loop where the agent can run against a real PR, use real context, execute through the guarded runtime, generate evidence, and produce reviewable learning candidates that improve future runs per project.

## v1 Positioning

Agent QA v1 should be positioned as:

> A guarded QA runtime for PR-driven validation that converts project context into safe execution plans, runs them through `PlanExecutorService`, records evidence/reports, and generates reviewable project memory candidates for future improvement.

This is stronger than a one-off smoke runner because it closes the loop:

```txt
Real PR
-> demand/context
-> memory
-> scenarios
-> ExecutionPlan
-> PlanExecutorService
-> Evidence/Reports
-> learning candidates
-> reviewed project memory
-> better next execution
```

## What v1 Promises

- Runs the Agent QA flow end to end on a real PR without mocks.
- Reads real PR/diff context.
- Reads real ClickUp/task context when configured.
- Uses real project memory when available.
- Generates selected scenarios from demand, diff, and memory context.
- Generates or builds a guarded `ExecutionPlan`.
- Executes only through `PlanExecutorService`.
- Preserves Zod validation, semantic policy, preconditions, postconditions, quiescence, and business assertions.
- Produces run artifacts, reports, evidence, status, logs, and fallback outputs.
- Produces reviewable learning candidates from real execution.
- Keeps learned locators project-scoped and reviewable before they become memory.
- Avoids public tools for direct browser actions such as `click`, `fill`, `press`, or `navigate`.

## What v1 Does Not Promise

- It does not guarantee perfect support for every frontend.
- It does not guarantee success on inaccessible, icon-only, canvas-heavy, or deeply custom Shadow DOM interfaces.
- It does not eliminate all LLM fallback paths.
- It does not allow the LLM to execute Playwright directly.
- It does not persist ephemeral `el_*` identifiers as stable locators.
- It does not automatically promote learning candidates into project memory without review.
- It does not replace QA review for high-risk flows.

## v1 Gates

v1 should only be considered closed when these gates pass:

- `PRJ-11324` runs the productive flow end to end without mocks.
- A real controlled PR is selected.
- A real ClickUp card or equivalent demand source is configured.
- Real secrets/configuration are validated by preflight.
- Real memory is consulted and its impact is recorded.
- Scenario selection runs from real PR/demand/memory context.
- `ExecutionPlan` generation/build records `planSource`, `fallbackReason`, and `fallbackWarning`.
- Execution runs through `PlanExecutorService`.
- Reports and evidence are generated.
- PR comment publishing works, or fallback local publishing is saved and documented.
- `learning-candidates` are generated from run artifacts.
- Learning candidates include `runId`, `scenarioId`, `taskId`, and `stepId`.
- Learning candidates separate confirmed locators from inferred suggestions.
- Sensitive data is masked.
- No ephemeral `el_*` is persisted as stable memory.
- At least one follow-up execution reuses reviewed memory and shows reduced fallback, reduced LLM calls, or improved locator resolution.
- Limitations and next steps are documented.

## Learning Candidates

The v1 learning loop should treat execution results as candidates, not automatic truth.

Learning candidates should capture:

- successful locator resolutions;
- mapping between `ExpectedOutcome`, `ExecutionStep`, and the real observed element;
- semantic targets produced by the LLM;
- deterministic resolver path used by `LocatorResolverService`;
- cases where token-overlap fallback resolved a target;
- cases where `ElementAvailabilityResolver` opened a container;
- cases where `replan` or `decide` was still needed;
- gaps where project memory did not help;
- suggested `semantic_locator` entries for `.agent-qa/memory.md`.

Learning candidates should not capture:

- credentials;
- tokens;
- raw cookies;
- unmasked emails when privacy config requires masking;
- transient `el_*` element ids as stable locators;
- arbitrary DOM dumps as memory.

## Suggested Task Ownership

### `PRJ-11450` - Consultar memória real

Should record:

- which memory chunks were returned;
- whether they influenced scenario generation, plan building, or execution;
- gaps where memory was missing or unhelpful;
- no new learning should be written in this step.

### `PRJ-11453` - Gerar ExecutionPlan

Should record:

- `planSource`;
- `fallbackReason`;
- `fallbackWarning`;
- semantic targets generated by task;
- whether the plan contains fragile, generic, or non-observable targets;
- confirmation that no `el_*` appears in the plan.

### `PRJ-11456` - Executar via PlanExecutorService

Should record:

- deterministic locator resolutions;
- semantic fallback resolutions;
- `replan` usage;
- `decide` usage;
- target resolution failures;
- confirmation that `PlanExecutorService` remained the final execution authority.

### `PRJ-11459` - Gerar learning candidates

This is the main task for project-specific learning.

Expected output:

- `learning-candidates.json` or `learning-candidates.md`;
- source references for each candidate;
- confidence/risk;
- separation between confirmed locator and inferred suggestion;
- sensitive data masked;
- no persisted `el_*`.

### `PRJ-11461` - Documentar limitações e próximos passos

Should consolidate:

- what worked in the real flow;
- what failed or degraded;
- which learning candidates were useful;
- which candidates were rejected and why;
- limitations around accessibility, locators, Shadow DOM, canvas, icon-only controls, and missing labels;
- follow-ups for `LocatorResolverService`, semantic memory, and LLM call reduction.

### Optional Follow-Up: Review and Promote Learning Candidates

Create a follow-up task if needed:

```txt
Review and promote learning candidates to project memory
```

Acceptance:

- candidates reviewed manually;
- approved entries added to `.agent-qa/memory.md`;
- rejected entries documented;
- no sensitive data persisted;
- no `el_*` persisted;
- next run shows measurable improvement or a documented neutral result.

## Release Statement

If the gates above pass, v1 can be described as:

> Agent QA v1 is a production-ready guarded QA loop for PR validation with evidence, reports, safe macro-tools, and reviewable project learning. It is designed to improve per project over time while preserving executor authority and preventing direct LLM browser control.

This is an honest v1: useful, demonstrable, safe, and extensible, without pretending to solve every possible frontend automatically.
